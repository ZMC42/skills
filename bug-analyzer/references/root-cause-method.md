# Root Cause Method

按需在以下场景读取本文件：

- 用户给了较多日志、较大 diff、多个 commit 或多个异常现象
- 你需要把“现象 -> 根因”写得更严格
- 你需要判断多个候选原因谁更可信
- 你需要先根据技术栈解释日志、线程模型、资源语义或框架行为

## 先确认技术栈

分析前至少确认以下信息中的核心项：

- 编程语言
- 主要框架
- 数据库或中间件
- 部署/运行环境

技术栈不是背景材料，而是解释证据的前提。例如：

- 同样的超时日志，在 Java 连接池、Node.js 事件循环、Go context timeout 下语义不同
- 同样的空值问题，在静态类型语言和动态语言中的触发路径不同
- 同样的并发异常，在线程模型、协程模型、消息队列消费模型中的根因不同

如果技术栈缺失，只能给受限结论，并明确说明限制。

## 证据链写法

报告里尽量按这条线组织：

1. 用户表述：用户认为发生了什么
2. 校正后的现象：结合日志、代码、diff 后真正成立的现象定义
3. 技术栈语义：相关语言、框架、依赖或运行时约束是什么
4. 触发条件：在什么输入、状态、时序或环境下出现
5. bug 版本证据：`bug commit` 对应版本里，哪段旧代码、旧日志或缺失分支支撑该判断
6. 直接触发点：哪一行代码、哪个条件、哪个调用报错或偏离预期
7. 深层原因：为什么该状态会成立，为什么系统允许它传播到触发点
8. 修复机制：`fix commit` 如何切断原始链路
9. 剩余风险：哪些相邻路径仍可能失败

不要跳步。尤其不要把“报错位置”直接写成“根因”。
也不要把用户的第一句话直接当作最终问题定义。

每条关键证据至少满足以下之一：

- 有 `bug commit` 对应版本的代码片段支撑
- 有该版本运行时日志、堆栈或监控信号支撑

如果一条结论只有 fix 侧 patch、没有 bug 侧事实，它最多算“修复意图推断”，不能直接写成已确认根因。

## 先校正问题定义

用户常见的描述偏差包括：

- 把报错位置说成根因
- 把下游失败说成上游 bug
- 把偶发条件说成稳定复现
- 把修复后的现象改善误认为根因已经消失
- 把多个独立问题混成一个问题

当发现偏差时，用下面结构收敛：

1. 用户表述：原始说法是什么
2. 证据冲突：日志、代码、diff 与该表述哪里不一致
3. 修正定义：更准确的问题应该如何表述

修正定义必须更具体，至少包含以下之一：

- 真正的触发条件
- 真正失败的模块或状态
- 真正受影响的行为，而不是二级后果

## 三层 Why

对主问题至少回答三层：

1. 为什么这里报错或行为异常？
2. 为什么这个异常状态会进入这里？
3. 为什么系统之前没有阻断或纠正这个异常状态？

只有当第三层已经落到明确的工程原因时，才算接近根因，例如：

- 契约没有定义空值语义
- 状态机缺少迁移保护
- 重试逻辑破坏幂等性
- 资源释放依赖人工调用而非结构化生命周期
- 修复只在结果层兜底，没有改源头状态

## 大 Diff 收敛

当 diff 很大时，按下面顺序缩小范围：

1. 优先用测试步骤和 BUG 时间戳定位最相关的日志窗口
2. 优先命中日志中的文件名、函数名、错误关键词
3. 先看 fix commit 中哪些文件新增了判空、条件修正、锁、重试、回滚、释放资源
4. 回到 bug 版本核对这些点在旧逻辑中如何缺失或失效
5. 过滤纯重构、命名、格式化、注释、测试数据改动
6. 优先关注入口附近、状态流转节点、错误处理分支
7. 如仍无法收敛，明确给出“候选原因 A/B/C”并按证据排序

## 多异常现象

若用户给出多个现象，先做分流：

- 是否同一请求链路上的连续后果
- 是否同一状态源头在不同位置的不同表现
- 是否完全无关，只是同时出现

只有共享同一触发链或同一错误状态时，才合并为单一根因。

## 常见根因地图

- 空值/未初始化：前置契约不明、默认值缺失、异常路径跳过初始化
- 边界错误：空集合、单元素、最后一次循环、分页边界、时间边界
- 状态机错误：漏迁移、重复迁移、逆序迁移、失败态未回滚
- 并发时序：竞态、ABA、双写、重入、重复消费、超时与取消竞争
- 资源问题：连接/句柄未释放、池耗尽、缓存污染、对象复用越界
- 契约问题：接口字段语义变化、配置不一致、类型/格式假设不一致
- 错误处理问题：异常吞掉、错误码映射错误、回退逻辑覆盖真实失败

## 事实与推断

结论按证据强度标记：

- 已确认：可被日志、代码、diff 直接支持
- 高可信推断：证据链基本闭合，但缺一个运行时事实
- 待验证：存在多个候选原因，尚不足以下单一结论

如果“用户表述”本身不成立，也要把这件事标成已确认事实，而不是轻描淡写带过。
如果缺测试步骤或 BUG 时间戳，默认降低日志侧结论的证据强度，除非用户给出的日志片段本身已经完整且时间边界明确。
