# Report Templates

按需在以下场景读取本文件：

- 用户要正式复盘报告或长文档
- 你需要输出精简版模板
- 你需要给信息缺失时的标准回复

## 短版报告

```markdown
# Bug Analysis

## Symptom
- 用户表述：
- 校正后的现象定义：
- 测试步骤：
- BUG 时间戳：
- 触发条件：
- 影响范围：

## Evidence
- 技术栈：
- 版本语义：`bug commit` 为复现版本 / 引入版本待验证
- 日志关键点：
- bug 版本代码证据：
- fix commit 关键改动：
- 证据强度：已确认 / 高可信推断 / 待验证

## Root Cause
- 直接触发点：
- 深层原因：
- 证据链：

## Fix Assessment
- 修复做了什么：
- 是否覆盖根因：
- 剩余风险：

## Conclusion
- 一句话结论：
```

## 长版复盘

```markdown
# Bug Postmortem

## Background
- 背景：
- 影响面：

## Symptom Timeline
- 用户最初描述：
- 修正后的问题定义：
- 测试步骤：
- BUG 时间戳与日志窗口：
- 触发时间线：
- 日志/监控信号：

## Investigation
- 技术栈与运行环境：
- 版本锚点说明：`bug commit` 为复现版本，非引入版本
- 关键日志：
- bug 版本代码与日志证据：
- fix commit 关键改动：
- 调用链：
- 证据强度分层：

## Root Cause
- 直接触发点：
- 三层 why：
- 根本工程原因：

## Fix Review
- 修复方案：
- 修复原理：
- 为什么有效或为什么不完整：

## Risks And Follow-ups
- 回归风险：
- 建议补充的测试/监控/防御：
```

## 信息缺失模板

```text
当前还不能给出可靠根因，缺少以下关键证据：
1. [测试步骤：最小复现路径]
2. [BUG 时间戳：至少一次异常出现的精确时间点和时区]
3. [缺失日志/堆栈或日志文件路径]
4. [技术栈：语言、框架、中间件、数据库、运行环境]

这些信息会直接影响：
- [受影响的判断 1]
- [受影响的判断 2]
```

## 表述修正模板

```text
用户当前将问题表述为：[原始表述]

但从现有证据看，这个表述不够准确：
- [证据 1]
- [证据 2]

更准确的问题定义应为：
[修正后的现象/触发条件/影响范围]
```

## 多候选原因模板

```markdown
当前存在多个候选根因，按证据强弱排序：

1. 候选 A
- 支持证据：
- 仍缺失的验证：

2. 候选 B
- 支持证据：
- 仍缺失的验证：
```

## 首轮对齐模板

```text
当前我已经拿到的证据：
- [已知事实 1]
- [已知事实 2]
- [bug commit 只是复现版本，是否需要继续追溯引入 commit 待确认]

要继续做高置信根因分析，还缺：
1. [测试步骤]
2. [BUG 时间戳]
3. [最关键缺失项]

如果你先给我这些，我会先用时间点收敛日志，再用 bug 版本代码和 fix commit 改动闭合证据链，而不是泛泛猜测。
```
