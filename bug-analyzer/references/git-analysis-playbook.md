# Git Analysis Playbook

按需在以下场景读取本文件：

- 用户提供了 bug commit 和 fix commit
- 用户只给了 diff、commit message 或 PR 片段
- 你需要从大量改动中定位真正相关的修复点
- 你需要解释“为什么这个改动能修复这个 bug”
- 你需要区分“复现 bug 的版本”和“引入 bug 的提交”

## 先定义版本语义

默认术语：

- `bug commit`：测试时确认存在 bug 的版本锚点
- `fix commit`：用户认为已修复该 bug 的版本锚点或提交
- `introducing commit`：真正把 bug 引入代码库的提交

这三者不能混用。大多数用户只提供前两者，而不是第三者。

如果没有额外历史证据，不要把 `bug commit` 直接当作 `introducing commit`。

## 分析顺序

默认按这个顺序收敛：

1. 用现象、测试步骤、BUG 时间戳、日志和技术栈确定最相关的模块边界
2. 先看 `fix commit` 自身哪些文件真的改变了控制流、状态、异常处理、资源管理或配置语义
3. 对这些改动建立“修复点 -> 试图阻断的旧行为 -> bug 影响”的映射
4. 回到 `bug commit` 对应版本，找到旧代码、旧日志或缺失分支作为证据支撑
5. 只有当单个 `fix commit` 无法解释修复原理时，再扩展到 `bug_commit..fix_commit`
6. 过滤测试、文案、命名、格式化、纯重构改动
7. 只引用最关键的代码或 diff 片段，不要堆整段 patch

注意：这个流程回答的是“修复版本相对带 bug 版本改了什么”，不自动回答“bug 最初是在哪个 commit 被引入的”。

## 文件筛选标准

优先关注这些变更：

- 条件分支新增或删除
- 判空、默认值、边界处理变化
- 锁、重试、事务、回滚、超时、取消相关改动
- 资源申请/释放、连接池、缓存、队列消费相关改动
- 配置读取、字段映射、序列化/反序列化相关改动
- 错误处理、异常传播、返回码转换相关改动

低优先级改动通常包括：

- 注释
- 命名调整
- 格式化
- 无行为变化的抽取函数
- 与症状无关的测试桩数据

## commit 范围判断

如果 fix commit 很大，不要默认整个 commit 都在修 bug。优先拆成：

- 真实修复
- 顺手重构
- 测试补充
- 无关改动

如果 bug commit 和 fix commit 之间跨越多个提交，优先找最接近症状消失的最小改动集合，而不是机械分析全部历史。

如果用户要求定位引入 bug 的 commit，需要单独说明这通常要做更早历史范围的回溯、二分或症状复现，不应和修复差异分析混为一谈。

## diff 到根因的映射

对每个关键 diff 片段，至少回答：

1. 这段改动改变了什么行为？
2. 在 `bug commit` 对应版本里，旧行为是如何成立的？对应代码或日志证据是什么？
3. 改变的行为与 bug 现象如何关联？
4. 它解决的是根因、触发条件，还是只是表层症状？

如果只能解释“报错不再出现”，但解释不了“异常状态为何不再成立”，通常还没到根因层。

## 没有仓库上下文时

如果用户只给了片段化 diff：

- 先基于 fix 侧局部代码解释可确认的行为变化
- 明确指出还缺 bug 版本对应的旧代码或旧日志，不能把 patch 自身当成完整证据链
- 明确指出缺少调用方、状态来源、配置来源或运行时上下文
- 给出最值得补充的文件、函数或日志关键词

## 输出建议

写法上优先用这种句式：

- `文件/函数 A` 在旧逻辑中会在 `[条件]` 下产生 `[异常状态]`
- `文件/函数 B` 在修复后新增了 `[校验/同步/释放/回滚]`
- 因此原始触发链被切断在 `[具体节点]`，而不是仅仅掩盖了 `[表面症状]`
