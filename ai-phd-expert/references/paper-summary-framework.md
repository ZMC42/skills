# 论文总结详细框架

当用户请求详细总结、拆解、分析学术论文时，优先采用下面的主框架。若用户只要“简版总结”，可压缩为每部分 2-4 句，但保留相同逻辑顺序。

## 1. Task

- 用尽可能形式化的方式描述论文要解决的问题。
- 推荐写法：先定义输入、输出、目标函数、约束条件、评价标准，再补充实际应用背景。
- 若适用，写成如下形式：
  - 给定：$x \in \mathcal{X}$、条件 $c \in \mathcal{C}$、资源/结构约束 $\Omega$
  - 目标：学习映射 $f_\theta: (\mathcal{X}, \mathcal{C}) \rightarrow \mathcal{Y}$ 或策略 $\pi_\theta(a\mid s)$
  - 优化目标：$\min_\theta \mathcal{L}(\theta)$ / $\max_\theta J(\theta)$
  - 在何种约束下：样本稀缺、噪声、维度高、分布偏移、在线决策、组合约束等
- 不要只说“提升性能”，而要明确“在什么任务、什么设定、什么约束下优化什么”。

## 2. Challenge

- 说明传统方法在解决该 Task 时为什么困难。
- 至少回答以下两点：
  - 之前方法的核心假设是什么，这些假设为什么在当前问题中失效？
  - 主要难点来自哪里：表示能力不足、优化困难、信息利用不充分、计算/存储代价高、泛化差、鲁棒性差、信用分配难、搜索空间爆炸、约束难满足等
- 若有多类基线，按“方法类别 -> 共性瓶颈 -> 本文切入点”组织。

## 3. Insight & Novelty

- 这是整篇总结的核心部分，必须严格区分 `Inspiration`、`Insight`、`Novelty` 三层，不要混写。

### 3.1 Inspiration

- 作者的 Insight 受到什么启发？
- 启发可以来自：
  - 某个失败现象或反例
  - 某类经典方法的优点/缺点
  - 领域知识、物理/几何/因果/优化结构
  - 其他任务中的成功机制迁移
- 尽量说清楚：是“观察到了什么现象”，还是“借鉴了什么机制”。

### 3.2 Insight

- 作者真正洞察到了什么？
- 必须回答：
  - 这个 Insight 是针对问题定义、数据性质、模型结构、训练目标、推理策略中的哪一层？
  - 它揭示了为什么旧方法不行，以及为什么某种新设计可能有效。
- 若有多个 Insight，逐条列出，并为每条标注对应 Inspiration。
- 推荐格式：
  - `Insight i`：......
  - `启发来源`：......
  - `作用层面`：问题建模 / 表示 / 架构 / 优化 / 推理 / 数据构造 / 训练策略

### 3.3 Novelty

- 说明作者为了支撑上述 Insight，具体提出了哪些创新。
- 必须区分创新类型：
  - 架构创新
  - 方法创新
  - 策略创新
  - 目标函数/训练机制创新
  - 数据构造/任务定义创新
- 对于每一个 Novelty，必须严格按以下格式描述：
  - `【创新点解决的问题】 -> 【受哪个 Insight 启发】 -> 【设计了什么创新点，尽可能具体描述】`
- 不要只写模块名；要说明该创新点如何嵌入整体流程、改变了哪些信息流/优化路径/决策方式。
- 如果论文声称多个贡献点但其中某些只是工程组合而非真实创新，要明确指出。

## 4. Experiment

- 说明论文如何验证其 Task、Challenge、Insight、Novelty。
- 至少覆盖：
  - 实验设置：数据集、任务设定、评价指标、基线、实现细节是否公平
  - 主要结果：哪些结论真正被支持，哪些只是弱支持
  - 机制验证：消融实验、可视化、误差分析、敏感性分析是否足以支撑 Insight
- 额外指出潜在混淆因素：调参预算、预训练资源、数据泄漏、统计显著性、是否只在少数 benchmark 上有效。

## 5. Potential Flaw

- 从研究延展性和问题边界角度，系统分析潜在不足。
- 必须回答以下三个问题：
  - 当前问题情境是否有局限？能否通过扩展架构或问题设定，处理更高维度、更多条件、更多约束、更强交互、更动态的情境？
  - 在当前情境下，如果数据具有某些不良性质，方法会遇到什么特别困难？例如：长尾、强噪声、缺失标签、分布偏移、伪相关、极端不平衡、非平稳、重尾、不可辨识等。
  - 在这些困难中，哪一类最值得继续深挖并写成下一篇 paper？说明为什么它同时具备“重要性 + 非平凡性 + 可研究性”。
- 这一部分不要泛泛而谈“未来可提升效率/精度”；要提出真正可能构成研究问题的 failure mode。

## 6. Motivation

- 总结作者想到这篇文章 general idea 的思考路径，尽量用“第一性原理 + 追问式问句”表达。
- 推荐写法：
  - 先指出旧方法最本质的矛盾
  - 再提出一个自然问题：`既然......，那可不可以尝试......？`
  - 最后给出本文 general idea 如何顺着这个问题自然出现
- 目标不是复述摘要，而是还原“一个研究者最合理、最容易想到本文 idea 的思维链条”。
- 如果可能，给出 2-4 个递进式问句，体现从问题本质走到方法设计的过程。

## 可选附加

- **复现要点**：关键实现细节、训练配方、可能踩坑点
- **一句话结论**：什么时候值得用、什么时候不建议用
- **开放问题**：可能的失败案例、理论缺口、数据/标注瓶颈

## 建议输出骨架

```markdown
1. Task

2. Challenge

3. Insight & Novelty
3.1 Inspiration
3.2 Insight
3.3 Novelty

4. Experiment

5. Potential Flaw

6. Motivation
```
